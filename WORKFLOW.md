# 저장소 운영 규칙

## 1. 기본 원칙

* 이 프로젝트는 웹 서비스 기준으로 운영한다.
* 작업 범위의 최상위 기준 문서는 `doc/plan.md`다.
* 작업이 끝나면 기록, 검증, 커밋, 푸시 순서를 지킨다.
* 기본 브랜치는 `main`이다.

## 2. 표준 작업 순서

1. 범위 변경이 생기면 먼저 `doc/plan.md`를 수정한다.
2. 코드와 문서를 변경한다.
3. `history.md`에 작업 내역을 짧게 기록한다.
4. 관련 테스트 또는 최소 실행 확인을 수행한다.
5. 검증이 통과하면 커밋한다.
6. 원격 `origin`이 연결되어 있으면 `main`에 푸시한다.

## 3. 푸시 규칙

* 푸시 단위는 “의미 있는 작업 완료” 기준이다.
* 테스트나 실행 확인 없이 바로 푸시하지 않는다.
* 검증 실패 시 푸시하지 않고, 실패 원인과 상태를 `history.md`에 남긴다.
* 원격 저장소 URL은 사용자가 제공한 GitHub URL을 `origin`으로 연결한다.

## 4. 커밋 메시지 규칙

아래 접두어 중 하나를 사용한다.

* `feat:` 기능 추가
* `fix:` 버그 수정
* `docs:` 문서 변경
* `chore:` 설정, 구조, 잡무
* `refactor:` 동작 변화 없는 구조 개선
* `test:` 테스트 추가 또는 수정

예시:

* `feat: add lottery generator api`
* `docs: add workflow and history rules`

## 5. history.md 작성 규칙

각 작업은 아래 형식을 따른다.

* 날짜/시간
* 작업명
* 변경 요약
* 검증 결과
* 다음 작업

짧고 명확하게 적는다. 파일 목록만 나열하지 말고, “무엇을 바꿨는지, 왜 바꿨는지, 무엇이 남았는지”를 중심으로 기록한다.

## 6. .env.example 규칙

* 새 환경변수가 필요하면 실제 `.env`가 아니라 `.env.example`을 먼저 갱신한다.
* 민감한 값은 절대 커밋하지 않는다.
* README 또는 관련 문서에 변수 용도를 함께 남긴다.

## 7. Definition of Done

아래 조건을 만족해야 작업 완료로 본다.

* 요구 범위가 `doc/plan.md`와 충돌하지 않는다.
* 변경 내용이 `history.md`에 기록되어 있다.
* 관련 검증 결과가 남아 있다.
* 커밋 메시지가 규칙을 따른다.
* 원격이 연결된 경우 `origin/main`에 푸시되었다.

## 8. 원격 저장소 연결

원격 URL이 주어지면 아래 흐름을 사용한다.

```bash
git remote add origin <github-url>
git push -u origin main
```
