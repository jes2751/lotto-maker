# Workflow v1.1

## 1. 기본 원칙

Lotto Maker Lab은 `로컬 작업 -> 검증 -> 사용자 승인 -> 푸시` 순서로 운영한다.

- 커밋과 푸시는 같은 의미가 아니다.
- `main`은 배포 브랜치다.
- 사용자가 요청하거나 배포 가능한 상태일 때만 `origin/main`에 푸시한다.

## 2. 역할

### 2-1. 기획자

- 범위 결정
- 우선순위 정리
- 제외 범위 명시
- `plan.md`, `Product_Spec.md` 기준 유지

### 2-2. 웹디자이너

- 화면 구조 정리
- 위계, CTA, 가독성 기준 유지
- `Design_Guide.md` 기준 유지

### 2-3. 개발자

- 코드 구현
- API와 UI 연결
- 테스트와 빌드 검증

### 2-4. 테스터

- 기능 흐름 점검
- 회귀 체크
- 퍼블리시 전 QA

## 3. 작업 순서

1. 기획 기준 확인
2. 디자인 기준 확인
3. 구현
4. 테스트/빌드
5. `history.md` 기록
6. 사용자 승인 시 커밋/푸시

## 4. 문서 우선순위

1. `plan.md`
2. `Product_Spec.md`
3. `Design_Guide.md`
4. `Technical_Spec.md`
5. `WORKFLOW.md`
6. `check_list.md`
7. `history.md`

구현이 문서와 다르면 먼저 문서를 갱신하거나 범위를 다시 확인한다.

## 5. 검증 규칙

### 5-1. 자동 검증

- `npm test`
- `npm run build`

### 5-2. 수동 검증

- 홈, 추천기, 회차 조회, 통계, 분석 허브 흐름 확인
- CTA, 링크, 정책 페이지 접근 확인
- 모바일 화면 확인

## 6. 기록 규칙

`history.md`에는 아래 4가지를 남긴다.

- 작업
- 변경
- 검증
- 다음

## 7. 커밋/푸시 규칙

- 로컬 검증 전 커밋하지 않는다.
- 커밋 후에도 자동 푸시하지 않는다.
- 사용자가 요청했을 때만 푸시한다.
- `main` 푸시는 곧 배포 반영이라는 전제로 판단한다.

## 8. Definition of Done

아래 조건을 만족하면 한 작업 묶음을 완료로 본다.

- 문서 기준과 구현이 맞다
- 테스트와 빌드가 통과한다
- 핵심 흐름 수동 확인이 가능하다
- `history.md`가 갱신됐다

## 9. 운영 메모

- OneDrive/Windows 환경에서는 `.next` 캐시 문제가 날 수 있다.
- 빌드가 캐시 문제로 깨지면 `.next`를 정리한 뒤 다시 확인한다.
- 문서나 UI 카피가 깨질 경우 즉시 UTF-8 기준으로 복구한다.
