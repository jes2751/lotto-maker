# Workflow v1.1

## 1. 문서 목적

이 문서는 Lotto Maker Lab 프로젝트의 작업 운영 규칙을 정의한다.  
현재 프로젝트는 `로컬 우선, 검증 후 반영` 원칙으로 운영한다.

## 2. 기본 원칙

- 먼저 로컬에서 작업한다
- 로컬 검증 전에는 푸시하지 않는다
- `main`은 사실상 배포 브랜치처럼 다룬다
- 사용자가 요청했을 때만 `origin/main`에 푸시한다
- 모든 중요한 작업은 `history.md`에 남긴다

## 3. 역할

### 3-1. 기획자

책임:

- 범위 확정
- 우선순위 결정
- 제외 범위 명시
- `plan.md`, `Product_Spec.md` 기준 관리

### 3-2. 웹디자이너

책임:

- 정보 구조 정리
- 시각 위계 설정
- 가독성과 정돈감 유지
- `Design_Guide.md` 기준 관리

### 3-3. 개발자

책임:

- 기획과 디자인 기준에 맞는 구현
- API, 데이터, UI 연결
- 테스트와 빌드 검증

### 3-4. 테스터

책임:

- 핵심 사용자 흐름 확인
- 회귀 여부 점검
- 퍼블리시 전 품질 확인

## 4. 표준 작업 순서

1. 기획 확인
2. 디자인 기준 확인
3. 로컬 구현
4. 로컬 테스트
5. `history.md` 기록
6. 커밋 후보 정리
7. 사용자 승인 후 푸시

## 5. 커밋/푸시 규칙

- `커밋`과 `푸시`는 분리한다
- 배포 가능한 상태가 아니면 푸시하지 않는다
- 사용자가 명시적으로 요청했을 때만 `origin/main` 푸시를 진행한다
- 중간 개발은 로컬 변경 상태로 유지할 수 있다

## 6. 문서 우선순위

1. `plan.md`
2. `Product_Spec.md`
3. `Design_Guide.md`
4. `Technical_Spec.md`
5. `WORKFLOW.md`
6. `check_list.md`
7. `history.md`

범위가 바뀌면 먼저 상위 문서를 고친 뒤 구현으로 넘어간다.

## 7. 검증 규칙

### 7-1. 코드 변경 시

- `npm test`
- `npm run build`

### 7-2. UI 변경 시

- 주요 화면 구조 확인
- CTA 위치 확인
- 모바일 깨짐 여부 확인

### 7-3. 퍼블리시 전

- 광고 비노출 조건 확인
- Analytics 동작 확인
- 메타데이터와 도메인 확인

## 8. 기록 규칙

`history.md`에는 아래 4가지를 남긴다.

- 작업
- 변경
- 검증
- 다음

너무 긴 서술보다 실제 판단에 필요한 정보만 남긴다.

## 9. Definition of Done

작업은 아래를 만족해야 `완료`로 본다.

- 문서 기준과 충돌하지 않음
- 로컬에서 동작 확인 가능
- 테스트/빌드 통과
- 사용자에게 설명 가능한 수준으로 정리됨

## 10. 현재 운영 원칙 요약

- 로컬에서 충분히 정리된 뒤에만 푸시한다
- `main` 푸시는 배포 행위로 간주한다
- 완성도와 안정성을 우선한다
