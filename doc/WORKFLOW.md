# 저장소 운영 규칙

## 1. 기본 원칙

* 이 프로젝트는 웹 서비스 기준으로 운영한다.
* 작업 범위의 최상위 기준 문서는 `plan.md`다.
* 개발 흐름은 `기획자 -> 웹디자이너 -> 개발자 -> 테스터 -> 커밋 후보 -> 승인 후 푸시` 순서를 따른다.
* `main` 브랜치 푸시는 배포와 연결되어 있으므로, 커밋과 푸시는 같은 단계로 취급하지 않는다.
* 기본 브랜치는 `main`이지만, 로컬 작업 단계에서는 푸시를 서두르지 않는다.

## 2. 운영 단계

### 로컬 작업 단계

* 코드와 문서를 먼저 로컬에서 수정한다.
* 의미 있는 작업 단위가 되기 전까지는 푸시하지 않는다.
* 필요하면 커밋도 미루고, 워크트리 상태로 검토를 이어갈 수 있다.

### 로컬 검증 단계

* `npm test`
* `npm run build`
* 필요 시 `npm run dev` 기준 브라우저 확인
* 실패 시 원인과 상태를 `history.md`에 남기고 로컬에서 계속 수정한다.

### 커밋 후보 단계

* 변경 범위가 어느 정도 묶였는지 확인한다.
* `history.md`에 작업, 변경, 검증, 다음 항목을 남긴다.
* 이 단계에서는 커밋까지만 할 수 있고, 푸시는 아직 하지 않는다.

### 배포 반영 단계

* 푸시해도 되는 상태인지 최종 확인한다.
* 사용자가 명시적으로 요청했을 때만 `origin/main`으로 푸시한다.
* 푸시 이후 실제 배포 URL에서 다시 확인한다.

## 3. 역할 정의

### 기획자

* 범위를 확정한다.
* 요구사항과 제외 범위를 문서로 남긴다.
* 제품 문서가 최신 상태인지 확인한다.

### 웹디자이너

* 화면 구조와 UI 방향을 정의한다.
* 공통 컴포넌트 규칙, 반응형 기준, 상태별 표현 기준을 문서로 남긴다.
* 기획 범위 안에서 화면 경험이 일관되게 보이도록 기준을 맞춘다.

### 개발자

* 기획과 디자인 기준 안에서 코드와 문서를 구현한다.
* 코드와 문서의 용어, 필드명, 구조, UI 반영 결과를 일치시킨다.
* 관련 테스트 또는 최소 실행 확인을 수행한다.

### 테스터

* 요구사항과 UI 기준으로 구현 결과를 검증한다.
* 회귀 여부를 확인한다.
* 테스트베드 검증을 포함해 실패 사항과 보류 이슈를 기록한다.

## 4. 역할별 산출물

### 기획자 산출물

* `plan.md`
* `Product_Spec.md`

### 웹디자이너 산출물

* `Design_Guide.md`
* 화면 상태, 반응형, 공통 UI 기준

### 개발자 산출물

* 코드 변경
* `Technical_Spec.md`와 관련 구현 문서 갱신
* `history.md` 작업 기록

### 테스터 산출물

* 테스트 결과
* 실패 원인
* 미해결 이슈 또는 보류 사항

## 5. 역할별 필수 게이트

### 기획자 게이트

* 범위와 제외 범위가 문서화되어 있다.
* 구현 대상 문서가 최신 기준과 충돌하지 않는다.

### 웹디자이너 게이트

* 주요 화면 구조가 정의되어 있다.
* 공통 UI와 반응형 기준이 문서화되어 있다.
* 로딩, 빈 상태, 오류 상태 표현 기준이 있다.

### 개발자 게이트

* 구현이 기획 범위와 디자인 기준을 벗어나지 않는다.
* 관련 문서와 코드가 동기화되어 있다.
* 관련 테스트 또는 최소 실행 확인 결과가 존재한다.

### 테스터 게이트

* 요구사항 기준 통과 또는 실패 상태가 명확하다.
* 실패 시 원인과 영향 범위가 기록되어 있다.
* 검증 결과가 `history.md`에 남아 있다.

## 6. 표준 작업 순서

1. 기획자: 범위와 제외 항목 확정
2. 웹디자이너: 화면 구조와 UI 기준 정리
3. 개발자: 코드와 관련 문서 수정
4. 개발자: `history.md` 기록
5. 개발자: 테스트 또는 실행 확인
6. 테스터: 요구사항과 UI 기준으로 검증
7. 검증 통과 후 커밋 후보로 정리
8. 사용자 승인 후 `origin/main` 푸시

문서 변경만 있는 경우:

* 기획자 게이트만으로 진행 가능하다.
* 웹디자이너, 개발자, 테스터 게이트는 `영향 없음 확인`으로 짧게 기록한다.

버그 수정인 경우:

* 개발자는 재현 원인과 수정 범위를 남긴다.
* 테스터는 회귀 여부를 함께 확인한다.

## 7. 커밋과 푸시 규칙

* `커밋`은 로컬 작업 정리 단계다.
* `푸시`는 배포 반영 단계다.
* 커밋했다고 바로 푸시하지 않는다.
* `main` 푸시는 배포 가능한 상태일 때만 수행한다.
* 사용자가 명시적으로 요청했을 때만 `origin/main`으로 푸시한다.
* 검증 없이 푸시하지 않는다.
* 원격 저장소 URL은 사용자가 제공한 GitHub URL을 `origin`으로 연결한다.

## 8. 브랜치 운영 기준

* 기본 브랜치는 `main`이다.
* 급하지 않은 개발 중간 상태는 로컬에서만 유지할 수 있다.
* 필요하면 별도 작업 브랜치를 사용할 수 있지만, 현재 기본 운영은 `로컬 작업 -> 검증 -> main 반영 여부 판단`이다.

## 9. 커밋 메시지 규칙

아래 접두어 중 하나를 사용한다.

* `feat:`
* `fix:`
* `docs:`
* `chore:`
* `refactor:`
* `test:`

## 10. history.md 작성 규칙

각 작업은 아래 형식을 따른다.

* 날짜/시간
* 작업
* 변경
* 검증
* 다음

작성 기준:

* 최소한 `작업`, `변경`, `검증`, `다음`은 항상 포함한다.
* 기획 변경, 디자인 변경, 구현 변경, 검증 결과가 드러나도록 적는다.
* 테스트 단계에서 발견한 실패나 보류 사항도 반드시 남긴다.
* 푸시하지 않은 상태라면 `로컬 반영만 완료` 같은 상태도 남긴다.

## 11. 배포 전 최소 확인

* `.env.example` 최신성 확인
* `npm install`
* `npm test`
* `npm run build`
* 필요 시 `npm run dev`
* `history.md` 기록 확인
* 푸시 범위가 배포 가능한 수준인지 확인

## 12. Definition of Done

* 요구 범위가 `plan.md`와 충돌하지 않는다.
* 기획자, 웹디자이너, 개발자, 테스터 게이트가 모두 통과했다.
* 변경 내용이 `history.md`에 기록되어 있다.
* 관련 검증 결과가 남아 있다.
* 커밋 메시지가 규칙을 따른다.
* 푸시가 필요한 작업인 경우에만 `origin/main`에 반영한다.

## 13. 원격 저장소 연결

```bash
git remote add origin <github-url>
git push -u origin main
```
