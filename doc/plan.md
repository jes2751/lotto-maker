# 로또 서비스 v1 플랜

## 1. 문서 체계

이 문서는 프로젝트의 최상위 기준 문서다. 상세 기준은 아래 문서를 따른다.

* `Product_Spec.md`: 제품 기획, 화면, 사용자 흐름
* `Design_Guide.md`: 웹디자인 기준
* `Technical_Spec.md`: 생성 로직, API, 데이터, 테스트 기준
* `WORKFLOW.md`: 역할, 작업 순서, 기록, 커밋/푸시 규칙
* `history.md`: 실제 작업 로그

기존 세부 문서는 `archive/`에 보관한다.

## 2. 서비스 방향

* 서비스 포지셔닝은 `혼합형`이다.
* 사용자의 첫 경험 중심은 `번호 생성기`에 둔다.
* 추천의 핵심 기준은 기존 당첨 데이터다.
* 통계와 회차 조회는 추천 결과가 어떤 과거 데이터 흐름을 참고했는지 이해시키는 보조 기능으로 제공한다.
* `당첨 보장`, `무조건 적중` 같은 표현은 금지한다.

## 3. v1 목표

* 외부 수집 자동화 없이도 즉시 실행 가능한 Next.js 14 App Router 골격 구축
* 정적 시드 데이터 기반으로 번호 생성, 회차 조회, 기본 통계 제공
* 기존 마지막 회차 이후 신규 당첨 데이터를 매주 일요일 자동 반영할 수 있는 운영 기준 정의
* 이후 Prisma/PostgreSQL 전환이 가능한 구조 준비

## 4. v1 범위

### 화면

1. 홈 `/`
2. 번호 생성기 `/generate`
3. 회차 조회 `/draws`
4. 기본 통계 `/stats`

### API

1. `GET /api/health`
2. `GET /api/v1/draws`
3. `GET /api/v1/draws/[round]`
4. `POST /api/v1/generate`
5. `GET /api/v1/stats/frequency`

### 생성 전략

* `frequency`
* `mixed`
* `random` 비교용 옵션

## 5. v1 제외 범위

* 로그인과 인증
* 생성 결과 저장
* 공유 링크
* 피드백 수집
* 다크 모드 토글
* FAQ, 소개, 운영자 페이지
* 상세 통계, 검색, 자동 수집 스케줄링

## 6. v1 이후 확장 로드맵

### 6-1. 추첨기 확장

* `필터 추첨기`: 제외수, 고정수, 홀짝, 연번, 배수 등 조건을 사용자가 직접 설정하는 방식
* `그룹 추첨기`: 번호를 여러 그룹으로 나누고 그룹별 선택 개수를 지정하는 방식
* `그룹 필터 추첨기`: 그룹 추첨기와 필터 조건을 결합한 고급 조합 방식
* `다양한 추첨기`: 합계, 표준편차, 연속 번호 수, 특정 패턴 제외 등 통계 기반 옵션 묶음
* `수동 추첨`: 사용자가 직접 번호를 선택해 조합을 완성하는 보조 기능
* `QR 코드 스캔`: 오프라인 로또 용지 정보를 읽어 저장/비교하는 기능
* `룰렛/이벤트형 추첨`: 재미 요소 중심의 보조 추첨 UX

### 6-2. 통계 확장

* `종합 통계`: 홀짝, 합계, 연번, 저고, 배수, 소수, 색상 등 여러 지표를 한 번에 보여주는 대시보드
* `번호별 출현 통계`: 번호별 누적 출현 횟수와 최근 흐름
* `연속 미출현 통계`: 특정 번호가 몇 회차 동안 나오지 않았는지 추적
* `홀수/짝수 통계`: 회차별 홀짝 분포 흐름
* `연속 번호 통계`: 연번 출현 빈도와 길이 분포
* `이월수/회귀 통계`: 이전 회차 번호가 다음 회차에서 다시 등장하는 패턴 분석
* `앞수합/뒷수합/첫수합/끝수합 통계`: 자리/구간별 합계 패턴 분석
* `AC`, `표준편차`, `궁합수`, `패턴`, `색상` 통계: 조합 특성 분석용 세부 지표

### 6-3. 가상 체험 기능

* `로또 시뮬레이션`: 가상 구매와 당첨 여부 시뮬레이션
* `1등 당첨 체험`: 재미 요소 중심의 결과 연출 기능

### 6-4. 확장 원칙

* v1 이후 기능은 `데이터 기반 추천`이라는 핵심 방향을 해치지 않는 범위에서만 추가한다.
* 이 서비스는 무료 서비스 기준으로 운영한다.
* 로그인 없이도 핵심 기능을 사용하는 방향을 기본값으로 둔다.
* 경쟁 서비스 기능을 그대로 복제하지 않고, 웹 사용성에 맞게 재설계한다.

## 7. 우선순위

1. 제품 기획과 화면 기획 확정
2. 디자인 기준 확정
3. 생성 로직과 API 계약 확정
4. 데이터 정책 확정
5. 주간 데이터 반영 운영 기준 확정
6. 구현과 검증

## 8. 성공 조건

* 핵심 문서 5개가 서로 충돌 없이 유지된다.
* `Product_Spec.md`만으로 사용자 흐름과 화면 상태를 이해할 수 있다.
* `Design_Guide.md`만으로 UI 기준을 재현할 수 있다.
* `Technical_Spec.md`만으로 개발자가 구현을 시작할 수 있다.
* `WORKFLOW.md` 기준으로 작업, 검증, 기록, 푸시가 관리된다.
