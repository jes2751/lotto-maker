# 기술 명세 v1

## 1. 기술 개요

* Next.js 14 App Router
* TypeScript
* Tailwind CSS
* 정적 시드 데이터 기반 런타임
* Prisma/PostgreSQL 전환 가능 구조
* 추천 로직의 기본 입력은 기존 당첨 데이터다.

## 2. 시스템 구조

* 프론트엔드: App Router 페이지
* 백엔드: Route Handlers
* 데이터 계층: 시드 저장소와 DB 저장소를 교체 가능한 구조

## 3. 타입과 인터페이스 기준

### 생성 전략

* `random | frequency | mixed`

권장 사용 기준:

* `frequency`, `mixed`는 데이터 기반 추천 전략
* `random`은 비교용 또는 참고용 전략

### 주요 필드명

* 생성 API 요청: `strategy`, `count`, `include_bonus`
* draw 응답: `round`, `drawDate`, `numbers`, `bonus`
* 생성 결과: `id`, `strategy`, `numbers`, `bonus?`, `reason`, `generatedAt`
* 스키마 메타 필드: `meta_json`

## 4. 생성 로직

### 공통 규칙

* 메인 번호는 1~45 범위의 6개
* 중복 없음
* 반환 전 오름차순 정렬
* 보너스는 메인 번호 제외 후 1개 선택
* 추천 전략의 입력 데이터는 시드에 저장된 기존 당첨 회차다.

### `random`

* 1~45에서 중복 없이 6개 무작위 추출
* 이유 문구: `완전 랜덤 기준으로 중복 없이 조합했습니다.`
* 역할: 데이터 기반 추천 결과와 비교하는 참고용 옵션

### `frequency`

* 시드 전체 회차, 즉 기존 당첨 데이터의 메인 번호 빈도를 기반으로 가중치 계산
* 기본 가중치 `1`, 등장 시 `+1`
* 이유 문구: `시드 회차에서 자주 나온 번호에 가중치를 둬 조합했습니다.`
* 역할: v1의 기본 데이터 기반 추천 전략

### `mixed`

* 3개는 `frequency`, 3개는 `random`
* 합친 뒤 오름차순 정렬
* 이유 문구: `빈도 가중치와 랜덤 조합을 섞어 균형 있게 구성했습니다.`
* 역할: 기존 당첨 데이터 기반 추천에 무작위 요소를 더한 확장 전략

## 5. API 계약

### `GET /api/health`

* 상태, 데이터 소스, 마지막 회차 날짜, 버전 반환

### `GET /api/v1/draws`

* 쿼리:
  * `limit`: 1~20, 기본값 `10`
  * `offset`: 0 이상, 기본값 `0`

### `GET /api/v1/draws/[round]`

* 존재하지 않으면 `404 NOT_FOUND`

### `POST /api/v1/generate`

* 요청:
  * `strategy`: `random | frequency | mixed`
  * `count`: 1~5
  * `include_bonus`: boolean

### `GET /api/v1/stats/frequency`

* 쿼리:
  * `period`: `all | recent_10`
  * `type`: `main`

### 공통 에러 형식

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "..."
  }
}
```

## 6. 주간 데이터 반영 로직

### 실행 주기

* 매주 일요일 1회 실행한다.
* 기준 시간대는 `Asia/Seoul`이다.
* 권장 실행 시간은 일요일 오전 배치다.

### 데이터 출처

* 기준 데이터는 동행복권 공식 당첨 결과 데이터다.
* 구현 전 공식 응답 경로를 확정하고 문서에 고정한다.

### 동작 규칙

1. 마지막 저장 회차를 조회한다.
2. 공식 최신 회차를 조회한다.
3. 신규 회차가 있을 때만 수집과 반영을 수행한다.
4. 검증을 통과한 경우에만 시드 또는 DB에 반영한다.
5. 반영 실패 시 기존 데이터는 유지한다.

### 실패 처리

* 실패 시 기존 마지막 정상 데이터는 유지한다.
* 오류 로그를 남긴다.
* 수동 재실행 가능한 구조를 유지한다.

## 8. 시드 데이터 정책

* 현재 데이터 소스는 `src/lib/data/seed-draws.ts`
* 각 회차는 `round`, `drawDate`, `numbers[6]`, `bonus`를 필수로 가진다
* 시드 변경은 수동 갱신으로 시작한다
* 추천 기능은 이 시드에 저장된 기존 당첨 데이터를 직접 참조한다

## 9. 검증 규칙

* 메인 번호와 보너스 번호는 모두 1~45 범위여야 한다
* 메인 번호는 중복되면 안 된다
* 메인 번호는 오름차순이어야 한다
* `bonus`는 메인 번호와 중복되면 안 된다
* 같은 `round`가 중복되면 안 된다

## 10. 테스트 완료 기준

필수 통과 기준:

* 홈: 히어로, 지난 회차 당첨번호 카드, 생성기 CTA가 모두 보여야 한다.
* 생성기: 기본 전략 `mixed`가 선택되고, 추천 성공/실패 상태가 모두 처리되어야 한다.
* 회차 조회: 최근 회차 목록이 보여야 하고 빈 상태/오류 상태가 정의대로 노출되어야 한다.
* 통계: 전체 기준과 최근 10회 기준이 정상 계산되어야 한다.
* API: `draws`, `generate`, `stats/frequency` 성공/실패 케이스가 모두 검증되어야 한다.
* 주간 데이터 반영 로직: 신규 회차 있음/없음/검증 실패 시나리오를 모두 확인해야 한다.

## 12. 광고 연동 기준

* 광고 스크립트는 `NEXT_PUBLIC_ADSENSE_CLIENT_ID`가 있을 때만 로드한다.
* 기본 인라인 슬롯은 `NEXT_PUBLIC_ADSENSE_SLOT_INLINE` 환경변수를 사용한다.
* 광고가 설정되지 않은 환경에서는 광고 슬롯을 렌더링하지 않는다.
* 광고 컴포넌트는 페이지 콘텐츠와 분리된 재사용 컴포넌트로 유지한다.
* 실제 배포 전에는 `ads.txt`와 퍼블리셔 ID를 운영 환경 기준으로 확정해야 한다.
