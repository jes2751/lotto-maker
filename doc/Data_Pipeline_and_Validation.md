# 데이터 수집 및 검증 정책 v1

## 1. 현재 데이터 소스

* v1 런타임 데이터 소스는 저장소 내부 정적 시드 배열이다.
* 기준 파일은 `src/lib/data/seed-draws.ts`다.
* v1에서는 외부 API 호출이나 자동 수집 스케줄러를 실행하지 않는다.

## 2. draw 데이터 최소 구조

각 회차는 아래 필드를 필수로 가진다.

* `round`
* `drawDate`
* `numbers`
* `bonus`

선택 필드:

* `totalPrize`
* `firstPrize`
* `winnerCount`

## 3. 검증 규칙

### 3.1 회차 단위

* `round`는 양의 정수여야 한다.
* `drawDate`는 ISO 형식의 날짜 문자열이어야 한다.
* `numbers` 길이는 정확히 6이어야 한다.
* `bonus`는 단일 숫자여야 한다.

### 3.2 번호 단위

* 메인 번호와 보너스 번호는 모두 1~45 범위여야 한다.
* 메인 번호 6개는 중복되면 안 된다.
* 메인 번호는 오름차순이어야 한다.
* `bonus`는 메인 번호와 중복되면 안 된다.

### 3.3 목록 단위

* 같은 `round`가 중복되면 안 된다.
* 기본 정렬 기준은 최신 회차 우선이다.
* 시드 데이터의 누락 회차가 발견되면 문서와 기록에 남기고 수동 보완한다.

## 4. 시드 데이터 관리 방식

* v1에서는 시드 배열을 수동으로 갱신한다.
* 시드 갱신 시 `doc/history.md`에 데이터 변경 사실과 검증 결과를 기록한다.
* 시드 변경 후 최소 확인 항목:
  * 최근 회차 수 확인
  * 번호 정렬 확인
  * 중복 회차 없음 확인
  * 보너스 번호 중복 없음 확인

## 5. DB 전환 원칙

* 런타임은 시드 기반으로 유지하되, 타입과 저장소 인터페이스는 DB 교체 가능 구조를 유지한다.
* Prisma 스키마는 `draws`, `generated_sets`를 중심으로 유지한다.
* 코드의 공용 필드명은 `drawDate`, `include_bonus`, `meta_json` 기준으로 문서와 일치시킨다.

## 6. 후속 자동 수집 전환 절차

후속 단계에서 아래 순서로 확장한다.

1. 공식 데이터 출처를 확정한다.
2. 수집 스크립트를 별도 모듈로 추가한다.
3. 원본 응답을 검증한 뒤 시드와 동일한 draw 구조로 정규화한다.
4. 수동 검증을 거쳐 DB 적재 또는 시드 갱신으로 반영한다.
5. 실패 시 마지막 정상 데이터 유지 원칙을 적용한다.

## 7. 테스트 기준

* 잘못된 번호 범위가 있으면 검증에서 실패해야 한다.
* 메인 번호가 정렬되지 않으면 실패해야 한다.
* 보너스가 메인 번호와 중복되면 실패해야 한다.
* 중복 회차가 있으면 실패해야 한다.
