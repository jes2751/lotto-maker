# Design Draft Examples v1.0

## 1. 목적

- 이 문서는 실제 구현 전에 디자인 방향을 합의하기 위한 시안 문서다.
- 대상은 홈, 생성기, 통계 허브의 상단 경험이다.
- 기준 문서는 아래 두 개를 따른다.
  - `doc/project/plan.md`
  - `doc/specs/Design_Guide.md`

## 2. 전제

- 제품 본체는 `통계 + 겹침 회피 + 판단 보조`다.
- 첫 3초에는 흥미가 필요하지만, 화면이 과도하게 게임처럼 보이면 안 된다.
- 따라서 시안은 아래 균형을 유지해야 한다.
  - 시작: 눌러보고 싶다
  - 중간: 아, 이건 데이터로 보는 서비스구나
  - 최종: 생각보다 믿고 볼 만하다

## 3. 시안 A: 데이터 컨트롤룸

### 한 줄 설명

- 가장 안정적이고 가장 신뢰형인 방향

### 홈 히어로 예시

- eyebrow: `과거 1등 데이터 + 우리 유저 데이터`
- title: `이번 주 번호, 공식 기록과 유저 흐름으로 같이 좁혀보세요`
- body:
  - `최신 회차를 확인하고, 공식 당첨 기록과 우리 유저 데이터를 나눠 보며 더 합리적으로 번호를 고를 수 있습니다.`

### 첫 화면 구조

- 좌측:
  - 메인 카피
  - CTA: `번호 생성하기`
  - 보조 CTA: `과거 1등 데이터`
- 우측:
  - 최신 회차
  - 번호
  - `회차 상세 보기`
- 하단:
  - `과거 1등 데이터`
  - `우리 유저 데이터`
  2분할 진입 카드

### 장점

- 가장 안전하다
- 제품 정체성이 쉽게 흐려지지 않는다

### 단점

- 첫 3초 훅이 아주 세지는 않다

## 4. 시안 B: 군중 회피 플레이보드

### 한 줄 설명

- 가장 추천하는 방향
- 흥미와 통계형 제품성이 함께 살아 있다

### 홈 히어로 예시

- eyebrow: `겹침 회피 플레이보드`
- title: `맞는 것도 중요하지만, 덜 겹치는 것도 중요합니다`
- body:
  - `과거 1등 데이터와 우리 유저 데이터를 나눠 보고, 지금 사람들이 몰릴 번호인지 덜 겹칠 가능성이 있는 조합인지 더 입체적으로 판단할 수 있습니다.`

### 첫 화면 구조

- 좌측:
  - 문제 제기형 카피
  - CTA: `지금 번호 뽑기`
  - 보조 CTA: `지금 군중 흐름 보기`
  - 짧은 상태 칩:
    - `과거 1등 데이터`
    - `우리 유저 데이터`
    - `덜 겹칠 관점`
- 우측:
  - 최신 회차
  - 이번 회차 대표 흐름
  - 다음 행동 3개:
    - `과거 1등 데이터`
    - `우리 유저 데이터`
    - `회차 해석 보기`

### 생성기 상단 예시

- title: `어떻게 뽑을지 먼저 고르세요`
- 전략 성격:
  - `빠르게`
  - `통계형`
  - `덜 겹치게`
  - `직접 조건 설정`
- 생성 후 연결:
  - `공식 당첨 흐름 보기`
  - `우리 유저 데이터 보기`

### 통계 구조 예시

- 1축: `과거 1등 데이터`
  - 공식 당첨 기록 기준
  - 장기/최근 흐름
  - 번호 상세
- 2축: `우리 유저 데이터`
  - 유저 생성 쏠림
  - 많이 생성된 번호
  - 전략 점유율

### 장점

- 왜 이 사이트를 써야 하는지가 바로 드러난다
- 단순 생성기와 차별화가 강하다
- 흥미와 신뢰가 같이 간다

### 단점

- 카피를 과하게 밀면 자극적으로 보일 수 있다

## 5. 시안 C: 실험실형 선택기

### 한 줄 설명

- 가장 새롭지만 가장 조심해야 하는 방향

### 홈 히어로 예시

- eyebrow: `실험실 모드`
- title: `이번 주, 당신은 군중 쪽입니까 아니면 비껴가는 쪽입니까`
- body:
  - `공식 기록과 실제 유저 흐름을 함께 보며, 더 대중적인 선택인지 덜 겹칠 선택인지 직접 판단해볼 수 있습니다.`

### 첫 화면 구조

- 좌측:
  - 질문형 카피
  - CTA: `3초 선택 시작`
- 우측:
  - 최신 회차
  - 유저 쏠림 신호
- 하단:
  - `대중형`
  - `균형형`
  - `비껴가기형`

### 장점

- 훅은 가장 강하다

### 단점

- 지금 단계에선 컨셉이 앞설 수 있다
- 잘못하면 도구보다 놀이처럼 보인다

## 6. 추천안

- 추천은 `시안 B: 군중 회피 플레이보드`

### 추천 이유

- `흥미`가 있다
- 하지만 `과장된 오락 서비스`로 보이지 않는다
- `과거 1등 데이터`와 `우리 유저 데이터` 두 축을 가장 자연스럽게 설명할 수 있다
- 생성기, 통계, 생성 통계 허브까지 같은 철학으로 이어 붙이기 쉽다

## 7. 추천안 기준 실제 문구 예시

### 홈

- title:
  - `이번 주 번호, 먼저 뽑고 남들이 몰릴 흐름까지 같이 보세요`
- body:
  - `공식 당첨 기록과 우리 유저 데이터를 나눠 보고, 더 대중적인 조합인지 덜 겹칠 조합인지 판단할 수 있습니다.`

### 생성기

- title:
  - `어떻게 뽑을지 먼저 고르세요`
- body:
  - `빠르게 시작할 수 있지만, 결과는 공식 당첨 흐름과 유저 생성 흐름으로 바로 비교할 수 있습니다.`

### 과거 1등 데이터 허브

- title:
  - `공식 당첨 기록에서 먼저 봐야 할 흐름을 잡아보세요`
- body:
  - `전체 회차와 최근 10회를 기준으로, 실제 당첨 기록에서 반복된 번호와 패턴을 먼저 읽습니다.`

### 우리 유저 데이터 허브

- title:
  - `우리 유저들이 이번 회차에 어떤 번호와 전략을 고르는지 한눈에 보세요`
- body:
  - `사람들이 실제로 어디에 몰리는지, 어떤 전략을 많이 쓰는지, 어떤 번호가 반복해서 등장하는지 공개 흐름으로 봅니다.`

## 8. 구현 전 체크포인트

- 홈 첫 화면에서 `문제 제기 -> 바로 생성 -> 데이터 진입`이 한 장면으로 읽히는가
- `/stats`와 `/generated-stats`가 이름만 다른 비슷한 화면으로 보이지 않는가
- 생성기가 `설정 화면`처럼 보이지 않고 `선택 화면`처럼 읽히는가
- 카피가 흥미를 만들되, 당첨 보장이나 예측기처럼 들리지 않는가

## 9. 현재 결론

- 현재 적용 기준 시안은 `시안 B`다.
- 홈, 생성기, 통계 허브의 다음 구현과 카피 수정은 모두 `군중 회피 플레이보드` 방향을 기준으로 한다.
- 단, 실제 구현에서는 아래 선을 넘지 않는다.
  - 과도한 게임성 금지
  - 운세/예측 톤 금지
  - 당첨 보장 암시 금지
  - 본체는 계속 `통계형 판단 보조 도구`
